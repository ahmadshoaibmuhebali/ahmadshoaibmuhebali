# KN02: Cloud Computing - Service-Modelle

## 🎯 Ziel

## 1. Teil-Challenge: Aufgabe 1 - Begriffe zuordnen

### Aufgabenstellung
Ordnen Sie die folgenden Begriffe dem entsprechenden Cloud-Service-Modell zu:
- Hosting von virtuellen Maschinen
- Nutzung von Google Docs für die Textbearbeitung
- Containerverwaltung mit Kubernetes

### Lösung

#### 1. Hosting von virtuellen Maschinen
**Cloud-Service-Modell: IaaS (Infrastructure as a Service)**

**Begründung:** Bei IaaS stellt der Provider die grundlegende Infrastruktur (virtuelle Maschinen, Speicher, Netzwerk) bereit. Der Benutzer hat direkten Zugriff auf diese Ressourcen und muss Betriebssystem, Middleware und Anwendungen selbst verwalten.

**Beispiele:** AWS EC2, Microsoft Azure Virtual Machines, Google Cloud Compute Engine

---

#### 2. Nutzung von Google Docs für die Textbearbeitung
**Cloud-Service-Modell: SaaS (Software as a Service)**

**Begründung:** Bei SaaS wird eine fertige Anwendung über das Internet bereitgestellt. Der Benutzer nutzt die Software über einen Webbrowser, ohne diese installieren zu müssen. Der Provider verwaltet alle technischen Aspekte (Sicherheit, Updates, Verfügbarkeit).

**Beispiele:** Microsoft 365, Slack, Salesforce, Jira, Dropbox

---

#### 3. Containerverwaltung mit Kubernetes
**Cloud-Service-Modell: CaaS (Containers as a Service) / CaaS wird oft auch unter PaaS subsummiert**

**Begründung:** Bei CaaS stellt der Provider eine Plattform zur Verwaltung und Orchestrierung von Containern bereit. Der Benutzer entwickelt Anwendungen in Containern, ohne sich um die zugrunde liegende Infrastruktur kümmern zu müssen. Kubernetes ist eine Container-Orchestrierungsplattform, die es dem Benutzer ermöglicht, Container zu deployen, zu skalieren und zu verwalten.

**Beispiele:** Google Kubernetes Engine (GKE), Amazon ECS/EKS, Microsoft AKS

## 2. Teil-Challenge: Aufgabe 2 - IaaS vs. PaaS

### Aufgabenstellung
Erklären Sie den Unterschied zwischen IaaS und PaaS in eigenen Worten. Welche Aufgaben übernimmt der Benutzer bei jedem Modell selbst?

### Lösung

#### Was ist IaaS?
**IaaS (Infrastructure as a Service)** stellt die unterste Ebene der Cloud-Services dar. Der Provider bietet grundlegende IT-Infrastruktur wie virtuelle Maschinen, Speicher und Netzwerkressourcen an. Der Benutzer mietet diese Ressourcen und zahlt nach Verbrauch.

**Aufgaben des Benutzers bei IaaS:**
- ✅ Auswahl und Verwaltung des Betriebssystems (Windows, Linux, etc.)
- ✅ Installation und Verwaltung von Middleware (Datenbanken, Web-Server, etc.)
- ✅ Installation und Verwaltung von Anwendungen
- ✅ Sicherheitspatches und Updates des Betriebssystems
- ✅ Verwaltung von Benutzerkonten und Zugriffskontrolle
- ✅ Backup und Disaster Recovery auf Anwendungsebene

**Aufgaben des Providers bei IaaS:**
- ✅ Bereitstellung der physischen Hardware
- ✅ Verwaltung des Rechenzentrums (Stromversorgung, Kühlung, Sicherheit)
- ✅ Netzwerkinfrastruktur und Sicherheit
- ✅ Virtualisierungstechnologie und Hypervisor-Management
- ✅ Sicherheitspatches für Hypervisor und Firmware

**Beispiele:** AWS EC2, DigitalOcean, Linode

---

#### Was ist PaaS?
**PaaS (Platform as a Service)** ist eine mittlere Ebene, die eine komplette Entwicklungs- und Ausführungsumgebung bereitstellt. Der Provider bietet nicht nur Infrastruktur, sondern auch Entwicklungs-Tools, Laufzeitumgebungen und Services an. Der Benutzer muss sich nicht um die Infrastruktur kümmern, sondern konzentriert sich vollständig auf die Entwicklung seiner Anwendung.

**Aufgaben des Benutzers bei PaaS:**
- ✅ Entwicklung und Programmierung der Anwendung
- ✅ Deployment der Anwendung
- ✅ Verwaltung der Anwendungsdaten
- ✅ Konfiguration der Anwendungsumgebung

**Aufgaben des Providers bei PaaS:**
- ✅ Bereitstellung der gesamten Infrastruktur
- ✅ Verwaltung des Betriebssystems und der Middleware
- ✅ Bereitstellung von Entwicklungs-Tools und IDEs
- ✅ Sicherheitspatches und Updates (vollständig)
- ✅ Skalierung der Ressourcen
- ✅ Backup und Disaster Recovery (auf Infrastrukturebene)

**Beispiele:** Heroku, Google App Engine, AWS Elastic Beanstalk, Firebase

---

### Vergleichstabelle: IaaS vs. PaaS

| Aspekt | IaaS | PaaS |
|--------|------|------|
| **Verwaltete Komponenten** | Nur Infrastruktur | Infrastruktur + Plattform + Tools |
| **Kontrolle** | Maximale Kontrolle über die Infrastruktur | Weniger Kontrolle, dafür mehr Komfort |
| **Komplexität** | Höher (Benutzer verwaltet mehr) | Niedriger (Provider übernimmt mehr) |
| **Flexibilität** | Sehr flexibel | Weniger flexibel, aber standardisiert |
| **Benutzergruppe** | System-Administratoren, Infrastruktur-Teams | Entwickler |
| **Betriebssystem-Wahl** | Frei wählbar | Vom Provider vorgegeben |
| **Skalierung** | Manuell oder semi-automatisch | Automatisch durch Platform |
| **Kosten** | Nutzungsbasiert | Nutzungsbasiert |

### Kriterium 2: Unterschiede zwischen IaaS und PaaS erklären ✅

**Kernaussagen:**

1. **IaaS** bietet Hardware-Ressourcen (virtuelle Maschinen, Speicher, Netzwerk). Der Benutzer verwaltet alles darauf installierte.

2. **PaaS** bietet eine komplette Entwicklungsplattform. Der Benutzer konzentriert sich nur auf die Anwendungsentwicklung.

3. **Der entscheidende Unterschied:** 
   - Bei **IaaS** trägt der Benutzer Verantwortung für Betriebssystem, Middleware und Anwendungen.
   - Bei **PaaS** trägt der Provider Verantwortung für Betriebssystem und Middleware; der Benutzer nur für die Anwendung.

4. **Automatisierung:** PaaS bietet automatische Skalierung, automatische Updates und verwaltete Services, während IaaS mehr manuelle Verwaltung erfordert.

```
├─────────────────────────────────────────────────────┤

│         (Eigene Server, On-Premises)                 │
└─────────────────────────────────────────────────────┘
```

---

## ✅ Zusammenfassung

- **IaaS** = Sie verwalten: Betriebssystem, Middleware, Anwendung
- **PaaS** = Provider verwaltet: Betriebssystem, Middleware; Sie verwalten: Anwendung
- **SaaS** = Provider verwaltet alles; Sie nutzen nur die Anwendung
- **CaaS** = Provider verwaltet die Container-Orchestrierung; Sie deployen Container

Diese Unterscheidung hilft bei der Wahl des richtigen Cloud-Service-Modells für Ihr Projekt!

# B) Infrastructure as Code (IaC)

## 🎯 Ziel
Sie haben ein grundlegendes Verständnis von **Infrastructure as Code** (IaC), kennen die **wichtigsten Vorteile und Tools** von IaC und können in einfachen Worten erklären, wie Docker und Kubernetes dabei helfen, Systeme zu verwalten.

## 2. Teil-Challenge: Aufgabe 1 - Automatisierung vs. Manuelle Konfiguration

### Aufgabenstellung

**1. Warum ist es besser, IT-Systeme automatisch zu konfigurieren statt manuell?**

**2. Was bedeutet Infrastructure as Code (IaC)?**

### Lösung

#### 1. Warum ist es besser, IT-Systeme automatisch zu konfigurieren statt manuell?

Automatische Konfiguration bietet mehrere entscheidende Vorteile:

**✅ Effizienz und Zeitersparnis:**
- Automatisierte Prozesse führen zu schnelleren Bereitstellungen (Minuten statt Tage/Wochen)
- IT-Personal kann sich von Routinearbeiten befreien und sich auf strategische Aufgaben konzentrieren
- Wiederholende, fehleranfällige Arbeiten werden eliminiert

**✅ Konsistenz und Zuverlässigkeit:**
- Manuelle Konfigurationen sind fehleranfällig und inkonsistent
- Automatisierung gewährleistet, dass jedes System identisch konfiguriert wird
- Reduziert menschliche Fehler um bis zu 90%

**✅ Skalierbarkeit:**
- Manuelle Verwaltung ist auf wenige Systeme begrenzt
- Mit Automatisierung können Tausende von Systemen gleichzeitig verwaltet werden
- Elastische Skalierung ermöglicht schnelle Reaktion auf verändernde Anforderungen

**✅ Wartung und Updates:**
- Automatische Updates und Patches werden zentralisiert durchgeführt
- Schnelle Behebung von Sicherheitslücken
- Rollback bei Problemen ist automatisiert möglich

**✅ Kostenersparnis:**
- Weniger Arbeitszeit pro System
- Weniger Ausfallzeiten = weniger Verluste
- Bessere Ressourcennutzung durch Optimierung

---

#### 2. Was bedeutet Infrastructure as Code (IaC)?

**Infrastructure as Code (IaC)** ist ein Ansatz, die IT-Infrastruktur durch Programm-Code und Konfigurationsdateien zu definieren und zu verwalten, statt sie manuell zu konfigurieren.

**Kernidee:**
- Infrastruktur (Server, Netzwerke, Speicher, Datenbanken) wird wie Anwendungscode behandelt
- Alle Infrastruktur-Definitionen liegen in **Versionskontrollsystemen** vor (z.B. Git)
- Änderungen sind **nachvollziehbar und auditierbar**
- **Wiederherstellung** ist schnell möglich (Disaster Recovery)

**Praktisches Beispiel:**
```
Statt: Manuell SSH-Zugriff auf Server → Installation durchführen

Nutzen Sie: IaC-Datei (z.B. Terraform)
↓
terraform apply
↓
10 neue Server mit identischer Konfiguration in 5 Minuten bereit
```

**Praktiken von IaC:**

| Praktik | Beschreibung |
|---------|-------------|
| **Versionsverwaltung (VCS)** | Git, GitLab, GitHub - Alle Infrastruktur-Änderungen sind dokumentiert |
| **Continuous Integration (CI)** | Neue Infrastruktur-Änderungen werden automatisch getestet |
| **Continuous Delivery (CD)** | Getestete Änderungen werden automatisch in Produktion übernommen |
| **Idempotenz** | Mehrmaliges Ausführen hat das gleiche Ergebnis (sicher!) |
| **Dokumentation** | Code ist selbstdokumentierend - Was und Warum sind klar |

**Vorteile von IaC:**

✅ **Geschwindigkeit** - Infrastruktur in Minuten bereit  
✅ **Konsistenz** - Identische Umgebungen (Entwicklung, Test, Produktion)  
✅ **Zuverlässigkeit** - Weniger menschliche Fehler  
✅ **Wartbarkeit** - Einfache Anpassungen und Erweiterungen  
✅ **Wiederherstellung** - Schnelle Wiederherstellung nach Ausfällen  
✅ **Kosten** - Optimierte Ressourcennutzung  

**Use-Case Beispiel:**
Ein Unternehmen möchte eine neue Testumgebung für ein Projekt aufbauen:

❌ **Manuell:** 
- IT-Team muss 20-30 Server einzeln konfigurieren
- Dokumentation führen
- Fehlerquellen = hohe Ausfallrisiken
- **Dauer: 2-3 Wochen**

✅ **Mit IaC:**
- Eine Konfigurationsdatei definiert alle 20-30 Server
- `terraform apply` startet die automatische Bereitstellung
- Identische Konfiguration garantiert
- **Dauer: 30-60 Minuten**

## 2. Teil-Challenge: Aufgabe 2 - Tools von IaC

### Aufgabenstellung

**1. Befassen Sie sich mit zwei Tools, die bei IaC verwendet werden (z.B. Docker, Terraform) und erklären Sie kurz, was diese tun.**

**2. Recherchieren Sie die Begriffe Kubernetes und Container-Orchestrierung. Fassen Sie in Stichworten zusammen, was damit gemeint ist und was Sie darunter verstehen.**

### Lösung

#### 1. Zwei wichtige IaC-Tools

##### **Tool 1: Docker**

**Was ist Docker?**
Docker ist ein Containerisierungs-Tool, das Anwendungen und ihre Abhängigkeiten in isolierte, portable Container verpackt.

**Wie funktioniert Docker?**

```
Klassische Bereitstellung:
Anwendung → Abhängigkeiten installieren → Betriebssystem-Unterschiede 
→ "Läuft bei mir" 😕

Docker:
Anwendung + alle Abhängigkeiten → Container (selbstständig, isoliert)
→ Läuft überall gleich ✅
```

**Praktisches Beispiel:**
```dockerfile
# Dockerfile definiert einen Container
FROM python:3.9
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
```

**Vorteile von Docker:**
- ✅ **Portabilität** - Container läuft überall (Laptop, Server, Cloud)
- ✅ **Isolation** - Jeder Container hat seine eigene Umgebung
- ✅ **Schnellstart** - Sekunden statt Minuten
- ✅ **Konsistenz** - Entwicklung = Produktion
- ✅ **Leichtgewichtig** - Weniger Ressourcen als virtuelle Maschinen
- ✅ **Versionierung** - Container-Images können getaggt und versioniert werden

**Anwendungsfall:**
Microservices-Architektur: Jede Service läuft in eigenem Container (Datenbank, API, Frontend)

---

##### **Tool 2: Terraform**

**Was ist Terraform?**
Terraform ist ein Infrastructure-Definition-Tool, das Cloud-Ressourcen (Server, Netzwerke, Datenbanken) deklarativ definiert und bereitstellt.

**Wie funktioniert Terraform?**

```
Terraform-Datei (.tf):
resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  tags = {
    Name = "Web-Server"
  }
}

Befehl: terraform apply
↓
AWS erstellt automatisch die definierte EC2-Instanz
```

**Praktische Beispiel-Konfiguration:**
```hcl
# main.tf - definiert Cloud-Ressourcen
provider "aws" {
  region = "eu-central-1"
}

resource "aws_instance" "app_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.small"
  key_name      = aws_key_pair.deployer.key_name

  tags = {
    Name = "AppServer"
    Environment = "Production"
  }
}

resource "aws_security_group" "allow_web" {
  name = "allow_web"
  
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

**Vorteile von Terraform:**
- ✅ **Deklarativ** - Sie definieren den Zielzustand, nicht die Schritte
- ✅ **Multi-Cloud** - Funktioniert mit AWS, Azure, Google Cloud, etc.
- ✅ **Wiederverwendbar** - Module können in Projekten wiederverwendet werden
- ✅ **Versionskontrolle** - .tf-Dateien in Git speichern
- ✅ **Plan-Funktion** - `terraform plan` zeigt Änderungen vor Ausführung
- ✅ **Idempotent** - Mehrmaliges Ausführen ist sicher

**Anwendungsfall:**
Bereitstellung einer kompletten AWS-Infrastruktur (EC2, RDS, S3, VPC) mit einer einzigen Terraform-Konfiguration

#### 2. Kubernetes und Container-Orchestrierung

##### **Was ist Kubernetes?**

**Kubernetes (K8s)** ist eine Open-Source-Plattform zur automatischen Verwaltung, Orchestrierung und Skalierung von Containern in Produktionsumgebungen.

**Stichwort-Zusammenfassung:**

| Aspekt | Erklärung |
|--------|-----------|
| **Container-Orchestrierung** | Automatische Verwaltung vieler Container auf mehreren Servern |
| **Deployment** | Automatische Verteilung von Containern auf verfügbare Ressourcen |
| **Skalierung** | Automatische Erhöhung/Senkung der Container-Anzahl basierend auf Bedarf |
| **Selbstheilung** | Automatisches Neustarten abgestürzter Container |
| **Load Balancing** | Verteilung des Datenverkehrs auf Container |
| **Storage-Verwaltung** | Persistente Datenspeicherung für Container |
| **Netzwerk-Verwaltung** | Automatisches Routing zwischen Container |
| **Rolling Updates** | Stufenweise Aktualisierung ohne Ausfallzeiten |

**Praktisches Szenario ohne Kubernetes:**
```
Probleme bei 100+ Containern manuell verwalten:
❌ Welcher Server? Wo ist noch Platz?
❌ Container stürzt ab? Wer startet ihn neu?
❌ Traffic steigt? Wer skaliert auf?
❌ Update nötig? Wie ohne Ausfallzeiten?
❌ Speicherplatz? Wer verwaltet die Volumes?
```

**Mit Kubernetes:**
```
✅ Automatische Platzierung auf Servern
✅ Self-healing: Container stürzt ab → automatischer Restart
✅ Auto-scaling: Last steigt → automatisch mehr Container
✅ Zero-downtime Deployments: Alte Container → Neue Container
✅ Storage: Automatische Verwaltung persistenter Daten
```

**Kubernetes Architektur - Komponenten:**

```
┌─────────────────────────────────────────────────────┐
│            Kubernetes Cluster                        │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Master Node (Control Plane):                        │
│  - API Server (Schnittstelle)                       │
│  - Scheduler (Plant Container-Placement)            │
│  - Controller Manager (Überwacht Zustand)           │
│                                                      │
│  ↓                                                   │
│                                                      │
│  Worker Nodes (ausführen Container):                │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐│
│  │ Worker Node 1│  │ Worker Node 2│  │ Worker Node│
│  │ (Pods)       │  │ (Pods)       │  │ (Pods)     ││
│  └──────────────┘  └──────────────┘  └────────────┘│
│                                                      │
└──────────────────────────────────────────────────────┘
```

**Grundbegriffe in Kubernetes:**

- **Pod** - Kleinste Einheit, enthält einen oder mehrere Container
- **Deployment** - Definiert wie viele Container-Replicas laufen sollen
- **Service** - Netzwerk-Zugang zu Pods (Load Balancing)
- **ConfigMap** - Speichert Konfigurationsdaten
- **Secret** - Speichert sensible Daten (Passwörter, API-Keys)
- **PersistentVolume** - Persistenter Speicher für Daten

**Praktisches Beispiel - Deployment:**
```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3  # 3 Container-Instanzen
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: web
        image: myapp:1.0  # Docker-Image
        ports:
        - containerPort: 8080
```

Was passiert:
1. `kubectl apply -f deployment.yaml`
2. Kubernetes startet 3 Container-Instanzen
3. Bei Traffic-Anstieg → automatisch mehr replicas
4. Container stürzt ab → automatischer Restart
5. Update auf Version 2.0 → Schrittweise Aktualisierung

**Use-Cases von Kubernetes:**

✅ E-Commerce: 1000+ Benutzer gleichzeitig → Automatische Skalierung  
✅ Streaming-Dienste: Millionen Zugriffe → Hohe Verfügbarkeit  
✅ Microservices: 50+ Services → zentrale Verwaltung  
✅ Multi-Cloud: Services auf AWS, Azure, Google Cloud gleichzeitig

# C) Globale Cloud-Plattformen

## 3. Teil-Challenge: Aufgabe 1 - Globale Infrastruktur Vergleich

### Aufgabenstellung
Erstellen Sie eine Tabelle, in der die globale Infrastruktur (Regionen und Verfügbarkeitszonen) von AWS, Azure und GCP verglichen wird.

### Lösung

#### Tabelle: Globale Cloud-Infrastruktur Vergleich

| Cloud-Plattform | Anzahl Regionen | Verfügbarkeitszonen | Besonderheiten |
|---|---|---|---|
| **AWS (Amazon Web Services)** | 31 Regionen | 99 Verfügbarkeitszonen | Marktführer; größtes globales Netzwerk; jede Region unabhängig betrieben |
| **Microsoft Azure** | 60+ Regionen | 300+ Verfügbarkeitszonen | Größtes Regionsnetzwerk; starker Hybrid-Cloud-Fokus; Best-for-Microsoft-Integration |

#### Detaillierte Erklärung:

**🔷 AWS (Amazon Web Services)**

```
Struktur:
31 Regionen weltweit

   └─ Mehrere Rechenzentren pro AZ (physisch getrennt)
```

- ✅ Meiste Regionen und Zonen
- ✅ Höchste Verfügbarkeit durch viele AZ

**🔷 Microsoft Azure**

```
Struktur:
60+ Regionen weltweit (größte Anzahl)
└─ 300+ Verfügbarkeitszonen
   └─ Hochverfügbarkeitsgruppen

Besonderheit: Hybrid-Cloud-Integration
- On-Premises + Cloud nahtlos verbunden
- Azure Arc: Verwaltung überall
- Azure Stack: On-Premises Azure
```

**Vorteile:**
- ✅ Meiste Regionen (60+)
- ✅ Hybrid-Cloud & On-Premises Integration
- ✅ Beste Microsoft-Integration (Windows, Office, etc.)
- ✅ Stark in Enterprise-Umgebungen

---

**🔷 Google Cloud Platform (GCP)**

```
Struktur:
38 Regionen weltweit
└─ 100+ Verfügbarkeitszonen
   └─ Multi-Zone-Placement

Besonderheit: Eigene Glasfaserinfrastruktur

- Optimiert für Datenverkehr & AI/ML
```

**Vorteile:**
- ✅ Sehr schnelles privates Netzwerk

## 3. Teil-Challenge: Aufgabe 2 - Dienste-Vergleich

### Aufgabenstellung
Wählen Sie je einen Dienst (z.B. Compute, Storage) aus AWS, Azure und GCP und erklären Sie, wie diese sich unterscheiden.

### Lösung

#### **Vergleich: Compute-Dienste** (Virtual Machines / Server)

##### **1. AWS - Elastic Compute Cloud (EC2)**

**Was ist EC2?**
EC2 ist AWS's Dienst für virtuelle Server (Computing-Ressourcen). Sie mieten Rechenkapazität und zahlen pro Stunde/Minute.

**Charakteristiken:**
- 🔹 **Größte Auswahl** an Instance-Typen (100+)
- 🔹 **Flexible Bezahlung:** On-Demand, Reserved Instances, Spot Instances
- 🔹 **Skalierung:** Auto Scaling Groups (automatisch Instanzen hinzufügen/entfernen)
- 🔹 **Performance:** Hohe Rechenleistung für spezialisierte Workloads
- 🔹 **Betriebssystem:** Windows, Linux, Mac
- 🔹 **Älteste Plattform:** Best Practice & großes Ökosystem

**Pricing-Beispiel:**
```
t3.small (1 CPU, 2 GB RAM):
- On-Demand: ~$0.0209/Stunde
- Reserved (1 Jahr): ~75% Rabatt
- Spot: ~90% günstiger (unterbrechbar)
```

**Praktischer Use-Case:**
```
E-Commerce Webshop:
- Standard: t3.medium Instanzen für regulären Traffic
- Black Friday: Auto Scaling erhöht auf 50+ Instanzen
- Danach: Automatisch auf 2-3 Instanzen reduziert
```

---

##### **2. Azure - Virtual Machines (VMs)**

**Was sind Azure VMs?**
Ähnlich wie AWS EC2, aber mit starker Microsoft-Integration.

**Charakteristiken:**
- 🔹 **Microsoft-Integration:** Perfekt für Windows/SQL Server/Active Directory
- 🔹 **Hybrid Capabilities:** Einfache Verbindung zu On-Premises Systemen
- 🔹 **Azure Marketplace:** Vorkonfigurierte Lösungen
- 🔹 **Betriebssystem:** Windows (native), Linux
- 🔹 **Größe:** Gutes Sortiment an Instanztypen
- 🔹 **Lizenzierung:** BYOL (Bring Your Own License) für Windows

**Pricing-Beispiel:**
```
B2s (2 vCPU, 4 GB RAM):
- On-Demand (Linux): ~$0.0536/Stunde
- Reserved (1 Jahr): ~30-70% Rabatt
- Hybride Nutzung: Wenn Windows Server lizenziert
```

**Praktischer Use-Case:**
```
Enterprise Umgebung:
- Active Directory Integration für Authentifizierung
- SQL Server Datenbanken auf Windows VMs
- Azure Site Recovery für Disaster Recovery
- Hybrid: Lokale Server + Azure VMs
```

---

##### **3. Google Cloud Platform - Compute Engine**

**Was ist Compute Engine?**
GCPs Virtual Machine Service mit Fokus auf Performance und Datenverarbeitung.

**Charakteristiken:**
- 🔹 **Custom Machine Types:** Konfigurieren Sie genau die Ressourcen Sie brauchen
- 🔹 **Preemptible VMs:** Sehr günstig (bis 80% Rabatt) - können unterbrochen werden
- 🔹 **Commitment Discounts:** Flexible Rabatte (1 oder 3 Jahre)
- 🔹 **Performance:** Sehr schnelle Netzwerk-Performance
- 🔹 **Integration:** Nahtlose Integration mit BigQuery, ML Engine
- 🔹 **Betriebssystem:** Linux, Windows, Debian, Ubuntu, etc.

**Pricing-Beispiel:**
```
n1-standard-4 (4 vCPU, 15 GB RAM):
- On-Demand (Linux): ~$0.19/Stunde
- 1-Jahr Commitment: ~25% Rabatt
- 3-Jahr Commitment: ~52% Rabatt
- Preemptible: ~70% günstiger
```

**Praktischer Use-Case:**
```
Data Analytics Projekt:
- Compute Engine für ETL (Extract, Transform, Load) Jobs
- Preemptible VMs für Batch-Processing (kosteneffizient)
- Integration mit BigQuery für Datenanalyse
- Machine Learning Modelle trainieren
```
